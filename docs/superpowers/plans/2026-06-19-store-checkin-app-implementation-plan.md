# 门店顾客签到 APP 实现计划

日期：2026-06-19

## 技术栈

- 前端应用：uni-app。
- 主要目标端：Android App。
- 次要适配端：安卓平板、安卓手机；后续可扩展 H5、小程序、iOS。
- 本地数据：App 端本地数据库，用于断网保存顾客、签到、奖励和领取记录。
- 备份能力：后续接入云端备份接口或 uniCloud。
- 语音能力：优先使用 App 端语音能力；识别失败时回退到手写或选择顾客。

## 阶段 1：项目基础和页面骨架

目标：搭好 uni-app 项目结构，跑通手机和平板基础布局。

任务：

- 创建 uni-app 项目。
- 配置底部导航：签到首页、顾客、排行榜、奖励。
- 创建设置页入口。
- 建立统一字号、按钮、弹窗、列表样式。
- 适配手机和平板宽度，保证主要操作区大而清楚。

验收：

- Android 模拟器或真机可打开应用。
- 四个主页面可切换。
- 首页语音签到按钮在手机和平板上都足够醒目。

## 阶段 2：本地数据模型

目标：先把断网可用的数据底座做稳。

核心表：

- customers：顾客档案。
- checkins：签到记录。
- reward_rules：奖励规则。
- reward_claims：奖励领取记录。
- backup_meta：备份状态。

任务：

- 封装本地数据库初始化。
- 实现顾客新增、搜索、停用。
- 实现签到记录新增、查询、撤销。
- 实现按时间段统计签到次数。

验收：

- 断网状态下可以新增顾客并签到。
- 关闭应用后重新打开，数据仍然存在。
- 能正确查询本月签到次数。

## 阶段 3：签到核心流程

目标：完成日常最高频流程。

任务：

- 首页实现语音签到入口。
- 实现语音识别结果到顾客匹配。
- 实现同名顾客选择。
- 实现未找到顾客时的新建并签到。
- 实现重复签到二次确认。
- 实现最近签到记录和撤销。
- 实现选择名字签到。
- 实现手写名字入口；第一版可先使用文本输入或系统手写输入法兜底。

验收：

- 已有顾客可在“说名字、确认”两步内完成签到。
- 未找到顾客可在同一流程新建并完成签到。
- 当天重复签到必须二次确认。
- 最近签到可以撤销。

## 阶段 4：排行榜

目标：让店长快速看到某段时间内谁签到最多。

任务：

- 实现本月排行榜。
- 实现上月排行榜。
- 实现自定义时间段排行榜。
- 按签到次数倒序排序。
- 同次数按最近签到时间更早者在前。
- 显示待领取奖励标识。

验收：

- 新签到后排行榜实时更新。
- 时间段切换后统计准确。
- 手机和平板上列表清楚可读。

## 阶段 5：奖励规则和领取记录

目标：支持次数达标奖励和排名奖励。

任务：

- 实现次数档位奖励配置。
- 实现排名奖励配置。
- 根据时间周期生成待领取名单。
- 实现标记已领取。
- 实现撤回领取状态。
- 在排行榜和奖励页显示待领取状态。

验收：

- 顾客达到次数档位后出现在待领取名单。
- 排名进入奖励范围后出现在待领取名单。
- 标记已领取后状态正确保存。
- 误标记可撤回。

## 阶段 6：备份、恢复和导出

目标：降低单设备使用的数据丢失风险。

任务：

- 设置页显示最近备份时间。
- 实现本地数据导出。
- 实现云端备份接口预留。
- 实现备份失败提示，不影响本地功能。
- 后续接入恢复流程。

验收：

- 店长能看到是否长期未备份。
- 备份失败不影响签到。
- 本地数据可导出用于人工恢复或迁移。

## 阶段 7：易用性和现场测试

目标：验证 50 多岁店长是否能自然使用。

任务：

- 检查所有主要按钮尺寸。
- 检查所有确认弹窗文案。
- 检查平板横屏和竖屏。
- 用 20 条模拟顾客和 100 条签到记录测试性能。
- 模拟断网、重复签到、同名顾客、误签到、误发奖。

验收：

- 新用户无需培训即可完成语音签到。
- 主要操作没有小字、小按钮、复杂路径。
- 异常流程都有清楚回退。
